## 三分钟现场演示

1. 执行 `start_traceguard.bat`，确认控制台显示服务地址。
2. 打开 `http://127.0.0.1:8000/`，先展示健康状态和模型已加载状态。
3. 上传固定 BigGAN 样例，展示 `label`、`fake_prob`、`risk_level`、热力图和可疑区域。
4. 切换到证据冲突样例，明确说明 `label=real` 与 `tamper_type=local_tamper` 可以并存，系统不会用局部证据静默覆盖全局判断。
5. 导出 HTML 报告，展示输入摘要、结论、证据和人工复核建议。

## 故障回退

| 故障 | 现场处理 |
|---|---|
| CUDA 不可用 | 用 `start_traceguard.bat --device cpu` 启动，并说明推理耗时会增加 |
| 权重未找到 | 将经校验的 `best.pth` 放到项目根目录或 `checkpoints/`，不使用其他权重替代 |
| 端口 8000 被占用 | 使用 `start_traceguard.bat --port 8080`，访问 `http://127.0.0.1:8080/` |
| 浏览器请求失败 | 先查 `/api/v1/health`；若服务无法恢复，切换到已导出的离线 HTML 报告 |
| 现场样例结论与讲稿不一致 | 以界面实际输出为准，不口头覆盖或改写模型结果 |

## 评委问答边界

- 为什么 `risk_score` 不等于 `fake_prob`：前者融合全局概率、伪影、可疑面积、区域数量和一致性，用于审核优先级；后者仅来自 Detector。
- 是否已经证明跨域提升 17% 以上：尚未。原始消融表和完整复现脚本未入库前，仅仅介绍机制，不把该数字写成已复核结果。
- 红框是否代表真实篡改区域：当前红框是工程解释证据；像素级定位精度仍需带标注数据定量评价。
- 是否已经完成风险阈值校准：尚未。当前阈值来自配置，正式最优审核策略需要独立验证集。
- 社交平台 Accuracy 为什么较高：平台分类集 fake:real 为 9:1，因此必须同时看 Macro F1、分类别 Recall 和 GenImage 成对退化，不能仅仅引用 Accuracy。
