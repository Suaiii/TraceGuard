版本：v1.0  
日期：2026-07-12  
适用范围：`E:\aNB\TECH\AI竞赛` 内的竞赛材料、FakeTrace 项目报告资产、数据集、模型、脚本与交付文件。

## 1. 工作区准口径

本目录是 2026 第二届大学生人工智能安全竞赛的作品工作区。当前项目报告主题为 **FakeTrace：多模态 AIGC 内容安全平台**；它是信息安全竞赛产品/项目报告，不是 ATADD 竞赛复盘。

开始任何工作前，按任务需要优先读取：

1. `开放式自主命题作品赛参赛指南.pdf`：参赛边界与提交要求。
2. `竞赛计划.md`：现有项目计划；若与当前用户确认的项目口径冲突，以用户确认的口径为准，并在交付前指出冲突。
3. `作品报告模板.docx`、`原创性说明.docx`：报告结构、格式与声明要求。
4. `instruct.md`：竞赛通知与时间信息。
5. `output/doc/`：当前可交付报告版本。
6. `dataset/`、`best.pth`：仅在数据、模型或实验可复现性相关任务中读取，禁止将它们当作普通文本材料全量扫描。

`去年的相关信安内容报告.docx` 仅可用于排版、章节节奏和格式观察。不得复用、改写、导入或混合其中的技术创新内容、项目主张、系统设计、实验结果、图表或领域叙事。

## 2. 资产分层

| 层级 | 路径或类型 | 规则 |
|---|---|---|
| 受控源文件 | `*.md`、脚本、模板、原创性声明、正式报告源稿 | 可纳入 Git；每次修改需说明用途与来源。 |
| 可交付物 | `output/doc/` 中经确认的 DOCX/PDF | 只保留可复核的正式版本；文件名应表达项目与版本，不覆盖上一版。 |
| 大型可再生成资产 | `dataset/`、`best.pth` | 不纳入 Git；用清单记录来源、版本、校验值、用途与生成命令。 |
| 临时与派生产物 | `tmp/`、渲染页图、中间提取文本、日志 | 不纳入 Git；任务完成后清理或归档到明确位置。 |
| 外部参考 | 去年报告、竞赛下发原件、外部数据说明 | 保留原始文件名与来源，不在其上直接编辑。 |

## 3. Git 真实边界

1. 当前目录是 `Suaiii/TraceGuard` 的 Git 工作树；远端默认分支为 `main`。
2. 所有 Git 命令必须显式限定到本工作区路径，例如：

```powershell
git -C E:\aNB\TECH\AI竞赛 status --short --branch
git -C E:\aNB\TECH\AI竞赛 add -- AGENTS.md docs/inventory/source-assets.md
```

3. 禁止在工作树根目录执行无范围确认的 `git add .`、`git commit -a` 或清理命令；它会误纳入本地竞赛材料、数据或模型。
4. 提交前必须查看 `git diff --cached --stat` 与完整 staged 文件列表，确认不存在数据集、模型、压缩包、密钥、临时产物或不在当前 Issue 范围内的文件。
5. 在有效远端、默认分支和协作规则未确认前，不得推送、改写历史、强推或删除分支。

## 4. 库存清单规范

库存管理的目标是让每项资产都能回答：它是什么、来自哪里、谁生成、用于什么、是否可入库、如何复现或替换。

维护以下清单，目录不存在时仅在实际需要时创建：

```text
docs/inventory/source-assets.md
docs/inventory/datasets-and-models.md
docs/inventory/deliverables.md
```

每条记录至少包含：

```text
名称
相对路径
类别
来源或生成方式
用途
版本或日期
大小与 SHA-256（大型数据/模型必填）
Git 状态：tracked | ignored | external-reference | pending-review
责任人与最近核查日期
```

新增、替换或删除资产时：先更新对应清单，再修改引用它的报告、脚本或说明；禁止只替换文件而不更新来源与版本信息。删除前先确认没有报告、脚本或交付物引用。

## 5. 数据集与模型管理

1. 当前 `dataset/` 中的压缩包与 `best.pth` 为大型本地资产，默认仅仅保存在工作区，不进入 Git 历史。
2. 若需要实验，先记录使用的数据子集、划分、随机种子、模型权重标识、阈值和评测脚本版本；结果必须能追溯到这些输入。
3. 不得将压缩包、权重、二进制缓存或绝对内部路径写入报告正文。报告应解释数据来源、验证集来源、指标定义、对比方法、实验目的、结果解释和局限。
4. ATADD 仅可作为音频验证数据集、验证场景或证据来源；不得写官方成绩、排行榜、排名或以 ATADD 为中心的参赛结论。
5. 不因文件过大而静默截断数据、日志、证据或模型输出。需要摘要时保留原始资产，并明确摘要的派生关系。

## 6. FakeTrace 报告约束

1. 报告围绕多模态 AIGC 内容安全平台的检测、证据与风险处置能力展开。
2. 技术章节必须包含逻辑小节、架构或流程图、设计动机、模块结构、实现细节及其对平台的支持关系；不得只扩写散文式说明。
3. 图表默认使用本地 `nature-figure` 的 Python 工作流生成。图内仅保留必要模块、箭头、坐标轴、图例和简短标注；图名与高层解释写在 Word 正文和题注中。
4. 测试段落必须说明数据来源、指标来源和定义、基线或对比方法、实验目的、观察到的结果、结果支持的结论与局限。不得以内部文件名、路径或“对应输出”代替实验叙述。
5. 所有外部数据、图表和结论应保留可追溯来源。未经验证的主张不得写成已完成结果。

## 7. 文档与交付流程

```text
确认任务与报告口径
  -> 盘点受影响资产与引用关系
  -> 修改源文件或生成脚本
  -> 更新库存清单
  -> 生成交付物
  -> 文本与视觉核查
  -> 限定路径的 Git 暂存与复核
```

DOCX/PDF 修改必须做视觉渲染核查；仅提取文本不足以证明版式正确。报告交付前检查标题、项目名称、图题、表题、引用、页码和声明的一致性。

---

## 8. 验收与交接

每项完成工作至少报告：

- 修改或新增的文件；
- 受影响资产及其 Git 状态；
- 已执行的验证；
- 未执行验证及原因；
- 剩余风险、待补信息或人工确认项。

代码审查和文档审查优先报告数据污染、误纳入大型资产、来源不可追溯、报告口径偏移、未验证结果和版式损坏等问题，而不是泛泛总结。
